C++ Coding Standards and Practices

  • 文件名

    • 每个源代码文件应include一个文件。每个include文件应描述单个类或紧密集成的类集。通用指南适用于包含（.h和.hpp）文件的包含定义和非实例化的文件。因此，包含文件可以在多个文件中使用。源文件（.C或.cpp）定义了在编译时将其加载到内存中的代码。它实例化包含文件中定义的定义。

      例如，一个类定义将包含在“ .hpp”包含文件中。全局变量将在“ .cpp”文件而不是“ .hpp”文件中实例化。这是因为如果多次使用了包含文件，则将多次定义全局变量。 “ extern”语句适用于包含文件，因为它不声明变量的内存，而只是识别其存在。

    • 文件名应反映类名。虽然类名通常以大写字母开头，但是包含文件和源文件通常以小写字母开头。即文件classBase.hpp将包含ClassBase {}的定义，而classBase.cpp将包含其功能的源代码。

    • 类名称应遵循以下格式：CXxClassID，其中“ Xx”表示组件。 （即Mn main，Fm文件管理器，Cp命令处理器）每个单词的第一个字符应大写。命名文件时，请不要使用“ C”前缀。

    • 当在另一个包含文件或源文件中引用一个包含文件时，请勿包含完整路径。这避免了由依赖于操作系统的引用目录路径的方法所造成的可移植性问题。使用编译器命令行标志包括路径。 （即-I /路径名）

    • 对于所有项目包含文件，请使用双引号（“”）；对于操作系统或编译器提供的包含文件，请使用尖括号（<>）。

    • 为了简化脚本的使用和制作文件，文件名（和目录路径）不应包含空格，括号“（”或大多数操作系统不支持的疯狂字符。

  • 文件格式

    • Command: dos2unix filename.cpp
    • In vim: :1,$ s/{ctrl-V}{ctrl-M}//
  • 文件头
    • 版权声明
    • 注释放在文件顶部

  • 头文件

    • 避免重复引用 #ifndef class_name_h/#define class_name_h

    • 减少头文件中#include文件数量，将减少构建时间。使用前置声明 + 指针（引用）。stl容器不允许前置声明。

    • C++调用C函数使用以下声明

    • 避免在“ .hpp”包含文件中使用“命名空间”。如果文件包含在源代码中，则程序员可能不再使用命名空间中定义的变量，并且可能会遇到变量名冲突。

  • 类和结构体

    • 对于类和结构定义，标准实践以大写字母开头。基类通常会在类名后附加后缀“ Base”。确保将虚拟（不是纯虚拟，因此不是“ = 0”）添加到基类析构函数中，以便同时调用这两个析构函数。

    • 示例类内容

  • 如果可能，应将与机器相关的代码放置在单独的文件中，以简化移植任务。
  • 函数
    • Names: Use complement names for complement functions: get/set, add/remove, create/destroy, start/stop, insert/delete, increment/decrement, begin/end, first/last, up/down, min/max, next/previous, open/close, show/hide, suspend/resume, etc.
    • 在“ .cpp”文件中列出函数的顺序与在“ .hpp”类声明中声明的顺序相同。

C++ variable naming convention and style

Prefix Description
m_ my_ Variable is a member of class. Some use the prefix “my“ for class member variables and “our“ for class member static variables. (Also “m” and “ms” for member and member-static) I have also seen the use of ““ as both a prefix and as a suffix to denote a class member variable. If using mixed languages remember that many FORTRAN compilers use a ““ suffix. Due to C++ name mangling there will probably be no conflict when using this notation. It is a common practice but theoretically could pose a problem with mixed languages. See Mixing FORTRAN and C YoLinux tutorial.
_ (underscore) Variable not intended to be exposed externally to the user. Internal workings only. Most often used by compiler and library authors.Sometimes used (not preferred) as a suffix to repressent a class member variable.Sometimes used as variable prefix for function arguments. When used with function arguments, it can represent in/out/inout:_variable: Function input variable. More explicit: _inVariablevariable_: Function output variable. More explicit: outVariable_variable: Input/output by function. More explicit: _inoutVariable_Note: Dangerous when mixing C with FORTRAN as FUNCTION subroutines as suffixed with “_” in the object code. See nm to view object symbols. See the g77/f77 compiler flag “-fno-underscore“.
__ Marks an use as extension to ANSI C++. Often not compiler independant. Usually reserved for use by compiler writers so it is best to avoid the double underscore.Note: Be careful when mixing with FORTRAN as “__” is often generated in FORTRAN object symbol names. See the f77 compiler flag “-fno-second-underscore“. Also used in Microsoft “Managed C++” directives.

C++ Code Review Checklist

代码同行评审应该寻找：

  • 标头：许可证或版权信息
  • 注释：使用doxygen标签注释函数，功能描述，输入/出参数
  • 遵循编码标准：变量命名，缩进，括号样式…
  • 使用标准API：不要重复发明和添加代码维护
  • 使用标准：日志文件格式标准，ISO或ANSI标准
  • 错误检查和处理。数据完整性检查：范围检查。
  • 资源泄漏：内存，数据库连接，文件，套接字……为每个逻辑路径（包括错误处理）释放。
  • 安全性：缓冲区利用，明文网络通信，….
  • 线程安全：全局变量。
  • 逻辑和控制结构：无限循环条件，正确的循环计数器，使用前初始化的变量，数学/算法，…
  • 性能：内存，浪费的操作，同步调用…
  • 回归测试，包括不良数据和错误。
